## Game Overview

N/A — Simple Todo App is a **mobile productivity application**, not a game. It is a lightweight task management tool for individuals to create, track, and complete daily tasks on mobile devices. The core interaction loop is: add a task → work through it → mark it complete → review progress.

---

## Problem / Opportunity

Individuals struggle to keep track of daily tasks without a simple, low-friction tool. Existing apps are often bloated with team features, subscriptions, or complex UX that overwhelms solo users. This app fills the gap for a **single-user, no-account-required, distraction-free task manager** — fast to open, fast to use, nothing more.

---

## Target Players

- **Busy Individual (Primary):** Ages 18–45, light-to-moderate smartphone user, wants to capture and clear tasks quickly without learning a complex tool. Values simplicity and speed over features.
- **Light Productivity User (Secondary):** Ages 16–60, uses their phone as their primary organisational tool, wants a clean list they can filter and tick off satisfyingly at the end of the day.

## Game Design

### Core Gameplay Loop

> *Adapted as: Core Interaction Loop*

1. User opens the app — lands directly on their task list (no login screen)
2. User creates a new task by entering a title and optional due date
3. User views all tasks in a chronological or filtered list
4. User marks tasks complete as they work through them
5. User deletes tasks no longer needed
6. User filters between complete and incomplete tasks to review progress
7. Session ends — state persists locally for next open

### Game Mechanics

> *Adapted as: App Mechanics*

- **Create Task:** Tap a "+" or "Add Task" button → input field for title (required, max [Assumed: 120 characters]) + date picker for due date (optional) → confirm to save
- **Mark Complete:** Tap a checkbox or swipe gesture on a task row → task visually marked as done (strikethrough text, greyed state) → moves to "Completed" filter view [Assumed: not auto-deleted]
- **Delete Task:** [Assumed: swipe-to-delete gesture with a confirmation prompt to prevent accidental deletion] → task permanently removed from local storage
- **View All Tasks:** Default list view on app launch showing all tasks sorted by due date ascending [Assumed], incomplete tasks first [Assumed]
- **Filter Tasks:** Toggle or tab control with two states — "Incomplete" and "Complete" — filters the visible list in real time without navigation change

### Progression System

N/A — No progression system. App state persists via local storage. [Assumed: No gamification in v1.]

### Win / Lose Conditions

N/A — No win/lose states. Success state for the user is a clear "Completed" list at end of day. [Assumed: An empty incomplete list could display a positive empty-state message, e.g. "All done for today! 🎉"]

## Features by Area

### Core Functionality
- Create a task with a required title field and an optional due date (date picker, no time selection in v1)
- Mark any task as complete/incomplete (toggleable)
- Delete any task (swipe-to-delete with [Assumed: confirmation prompt])
- View all tasks in a single scrollable list
- Filter task list by status: Incomplete | Complete
- Persist all task data locally between sessions (no data loss on app close)

### UI / UX
- **Task List Screen (Main Screen):** Full-screen list of tasks, filter toggle at top, FAB or top-right "+" button to add task
- **Add Task Sheet/Modal:** Slides up from bottom or presents as modal — title input, due date picker, Save + Cancel actions
- **Empty State:** Friendly empty-state view when no tasks exist or no tasks match the active filter [Assumed: different messages for each case]
- **Completion Visual State:** Completed tasks shown with strikethrough title and muted colour; checkbox shows filled/checked state
- **Delete Confirmation:** [Assumed: Swipe reveals red delete button or presents an alert — prevents accidental permanent deletion]

### Backend & Services
- **Local Database:** All task data stored on-device only (no backend, no internet required)
  - Task schema: `id` (UUID), `title` (string), `due_date` (nullable date), `is_complete` (boolean), `created_at` (timestamp)
- **No authentication required** in v1
- **No analytics, cloud sync, or API integrations** in v1

### Store & Deployment
- **iOS:** Submit to Apple App Store — requires Apple Developer account, App Store Connect setup, privacy policy (even with no data collection, Apple requires declaration), app icon in all required sizes
- **Android:** Submit to Google Play Store — requires Google Play Developer account, content rating questionnaire, privacy policy page/URL
- [Assumed: No third-party SDKs (no ads, no analytics) simplifies store compliance significantly]

## Open Questions

1. **Platform priority:** Should iOS or Android be built first, or is a cross-platform framework (Flutter/React Native) preferred to ship both simultaneously?
2. **Due date behaviour:** Should tasks with a past due date be visually flagged (e.g. shown in red)? Is a due date without a time sufficient, or will time-of-day be needed post-v1?
3. **Sort order:** Should tasks be sorted by due date, creation date, or manually reorderable? What happens to tasks with no due date — top or bottom of list?
4. **Completion behaviour:** Do completed tasks remain in the main list (filtered out by default) or move to a separate "Done" section? Can a completed task be unchecked/restored to incomplete?
5. **Delete confirmation:** Should deletion require a confirmation dialog, or is swipe-to-delete with an undo snackbar/toast preferred?
6. **Data persistence risk:** If the user deletes the app, all data is lost (no cloud backup in v1) — should this be communicated to users at first launch?
7. **Monetisation intent:** Is this a portfolio/utility project or is there a future commercial model planned? This affects decisions around analytics groundwork and architecture for v2.
8. **Design system:** Is there an existing design language or brand guide to follow, or should the design be defined from scratch?
